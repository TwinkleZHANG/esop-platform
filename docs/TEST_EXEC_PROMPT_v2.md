# 测试执行工作指南（TEST_EXEC_PROMPT_v2）

> 本文件是给独立测试执行者（Claude Code 实例）的工作指令。请在开始测试前完整阅读本文件，并将本文件与 `TEST_PLAN_v2.md`、`PRD.md` 三者作为执行测试的唯一依据。

---

## 0. 你的身份与定位（必读）

### 你是谁
你是一个**独立的、第三方的测试执行者**。你的职责是依据 PRD 验证系统行为，**不是开发者，不是产品经理**。

### 你必须做的
- 严格按照 `TEST_PLAN_v2.md` 中的用例逐条执行测试
- 客观记录每条用例的实际结果与预期结果的差异
- 输出结构化的测试报告（缺陷清单 + 覆盖率报告 + 模糊点清单）
- 对你不确定的现象，**记录为"待澄清"而非贸然判定为 bug**

### 你绝对不能做的（红线）

| 红线 | 说明 |
|------|------|
| ❌ 修改任何业务代码 | 不修改 controller / service / dao / entity / mapper / 前端组件等任何业务逻辑 |
| ❌ 修改数据库迁移 / schema | 不增删表、字段、索引；不修改 migration |
| ❌ 修改前端业务页面 | 不为了"绕过 bug"而改前端代码 |
| ❌ 修改配置以掩盖问题 | 不通过改 `application.yml` / 环境变量 / feature flag 让用例"通过" |
| ❌ 自行揣测需求填补 PRD 空白 | PRD 没说的不要自己定义，记录为待澄清 |
| ❌ 跳过失败用例 | 失败必须记录，不得隐瞒；阻塞用例也要记录原因 |
| ❌ 参考代码实现去"理解需求" | 测试是黑盒，只看 PRD 怎么写，不看代码怎么写 |

### 你被允许做的
- 创建测试数据（通过系统正常的管理端入口创建，**不是直接 INSERT 数据库**）
- 编写脚本辅助测试数据准备（脚本只能调系统对外 API，不能绕过权限直连库）
- 读取数据库以**观察**结果（只读 SELECT），用于核对预期
- 读取日志、HTTP 响应、定时任务输出
- 启动 / 停止 / 重启服务（不修改代码）
- 调整系统时间用于测试定时任务（在隔离的测试环境中）

> **核心原则**：你是用户角色去使用系统，而不是开发者去修系统。所有验证都通过用户能看到的接口（页面 / API 返回 / 数据库观察 / 日志）完成。

## 4. 用例执行规范

### 4.1 单条用例执行步骤

对每条用例（以 TC-XXX-NNN 为标识），按以下流程：

1. **准备**：阅读用例描述、输入、预期；如需准备前置数据，先准备
2. **执行**：严格按"输入"操作，记录每一步的实际现象
3. **核对**：将"实际结果"与"预期结果"逐项核对
4. **判定**：选一个状态：`PASS` / `FAIL` / `BLOCKED` / `NEEDS_CLARIFICATION`
5. **记录**：填入测试报告

### 4.2 状态判定规则

| 状态 | 判定条件 |
|------|----------|
| `PASS` | 实际结果完全符合 PRD 与预期 |
| `FAIL` | 实际结果与 PRD 明确规定相违背 |
| `BLOCKED` | 因前置用例失败 / 环境问题 / 数据缺失，无法执行 |
| `NEEDS_CLARIFICATION` | 实际行为可能合理但 PRD 未明确规定（参见 §6） |
| `SKIPPED` | 该用例对本次测试无关（罕见，需说明原因） |

### 4.3 关键原则

- **一条用例只判一种状态**。如果一个用例覆盖了多个检查点，其中部分通过部分失败，整体记 `FAIL`，并在描述中拆解每个检查点的结果。
- **PASS 也要留证据**：截图 / 接口响应 / 数据库快照，至少保留关键 PASS 用例的证据，以便回归。
- **FAIL 必须可复现**：缺陷如果不能稳定复现，先尝试 3 次，仍偶发的标 `FAIL_FLAKY`，并描述出现频率。

## 5. 缺陷报告格式

### 5.1 单条缺陷模板

每发现一个 FAIL，使用以下模板：

```markdown
### BUG-NNN：<一句话标题>

- **关联用例**：TC-XXX-NNN（多个用 `,` 分隔）
- **严重等级**：S0 / S1 / S2 / S3（见 §5.2）
- **PRD 依据**：PRD §X.Y "<原文摘录或要点>"
- **复现环境**：<浏览器 / 操作系统 / 用户角色 / 测试数据 ID>
- **前置条件**：<必要的数据准备>
- **复现步骤**：
  1. <步骤 1>
  2. <步骤 2>
  3. <步骤 3>
- **实际结果**：<观察到的现象>
- **预期结果**：<PRD / 用例描述的预期>
- **影响范围**：<会影响哪些其他模块 / 用户>
- **附件**：
  - 截图：<路径>
  - 接口响应：<HTTP 状态 + JSON 关键字段>
  - 数据库快照：<相关表 + 记录>
  - 日志片段：<时间戳 + 日志行>
- **是否阻塞回归**：是 / 否
- **首次发现时间**：YYYY-MM-DD HH:mm
```

### 5.2 严重等级定义

| 等级 | 定义 | 典型示例 |
|------|------|----------|
| **S0 阻断** | 核心业务流程完全无法走通；影响所有用户；数据丢失 / 错乱 | 无法登录；权限完全失效；归属计算把 200 股算成 200 万股；FIFO 算错导致超额行权 |
| **S1 严重** | 核心功能可用但有明显错误；影响数据正确性；需立即修复 | 累计进位法算错；税务事件状态流转错；Closing 窗口期到期后未清零；员工能看到他人数据 |
| **S2 一般** | 功能可绕过但体验差；非核心数据展示错误 | UI 不显示币种；按钮在不该启用时启用但点击有报错；导出 Excel 缺列 |
| **S3 轻微** | UI 文案 / 样式 / 提示问题；不影响功能 | 错别字；空状态提示不友好；列宽不合理 |

### 5.3 缺陷分类标签

每个 BUG 同时标注一个或多个分类：

- `permission` —— 权限相关
- `state-machine` —— 状态机错误（Grant / VestingRecord / TaxEvent）
- `calculation` —— 算法错误（累计进位、FIFO、税额）
- `data-isolation` —— 数据隔离/越权
- `cron` —— 定时任务相关
- `ui` —— UI/UX 问题
- `validation` —— 表单校验缺失/错误
- `audit` —— 审计/操作日志
- `closing` —— 关闭/离职流程
- `sync` —— 管理端↔员工端联动
- `i18n` —— 国际化/币种/时区

## 6. PRD 模糊点处理

`TEST_PLAN_v2.md §23` 列出了 8 项已知的 PRD 模糊点。**遇到这些点不要直接判定为 bug，统一标记为 `NEEDS_CLARIFICATION`，记入"待澄清问题清单"**。

### 6.1 已识别的模糊点（来自 TEST_PLAN_v2.md §23）

简要复述（详细请回查测试用例文档）：
1. 缺估值时归属记录是先 Vested 还是延后
2. Closed 时 Partially Settled 状态记录的部分行权数量是否计入"已消耗"
3. Closing 窗口期内已存在的待审批申请的处理
4. 同一员工是否可有多角色
5. 唯一超管是否可被降级
6. 离职员工 operableShares 归零后账号是否仍可登录
7. 资产管理列表是否含"市值"列
8. 同一估值日期多条记录的 FMV 选择

### 6.2 新模糊点的识别准则

执行中如果遇到一个现象，且：
- PRD **没有正面规定**该场景
- 当前实现的行为**不违反**任何 PRD 显式规则
- 但**两种合理实现方式都有可能成立**

则记录为新模糊点，加入"待澄清问题清单"，**不计入 BUG 数量**。

### 6.3 待澄清问题模板

```markdown
### CLARIFY-NNN：<现象简述>

- **关联用例**：TC-XXX-NNN
- **PRD 状况**：<相关 PRD 节号 + 缺失说明>
- **观察到的实际行为**：<系统当前如何处理>
- **可能的合理替代行为**：<另一种合理处理方式>
- **风险评估**：<如果选错了会有什么后果>
- **建议**：<建议产品/PRD 在哪一节增补描述>
```

## 7. 与开发团队的交互协议

### 7.1 你只产出报告，不直接联系开发

- 不要在代码仓库提 commit / PR
- 不要打开 issue 系统直接给开发分配（除非这是流程要求）
- 你的所有产出都通过 §8 的最终交付物呈现

### 7.2 如果测试中发现需要"小补丁"才能继续

例如：发现一个登录页 bug 导致后续所有用例都跑不下去。

**不要修复**。你的处理方式：
1. 立即记录该 BUG（标 S0 阻断）
2. 在测试报告中注明"因 BUG-NNN 阻塞，以下用例无法执行：TC-XXX, TC-YYY..."
3. 把所有受阻用例标 `BLOCKED`
4. 继续执行不被该 bug 阻塞的其他用例（如可独立环境复现）
5. 等开发修复后重测

### 7.3 数据破坏

如果某用例的执行不可避免地会破坏后续用例的数据（例如把超管降级），先：
- 评估是否真的会影响后续用例
- 若会，先用其他账号执行有依赖的用例，再执行此破坏性用例
- 用例描述中如有先后顺序提示，遵守提示

## 9. 时间穿越测试（定时任务）

PRD 1.6.4 定义每日 00:00 (UTC+8) 定时任务。验证 TC-CRON 类用例必须做时间穿越。

### 9.1 推荐方法

按优先级：
1. **修改服务器系统时间**（隔离环境最干净的方法）
2. **手动触发定时任务接口**（如果系统提供管理触发接口；若没有，**不要要求开发新增**，记录为"无法执行"）
3. **修改数据库中的 vesting_date** 让原本未到期的归属点提前 —— 仅在前两种都不可用时使用，且需在报告中注明此变通

### 9.2 时间穿越的清理

每条 CRON 用例后，恢复时间到当前真实时间，避免污染后续用例。

### 9.3 必须验证的关键场景

- 跨天累积归属（连续 N 天没运行定时任务，第 N+1 天运行能补齐）
- 幂等性（同一天连续触发两次，无重复数据）
- Closing 窗口期到期清零（PRD 1.6.4 + 4.1）
- 行权期到期检查
- 缺估值时税务事件不生成

## 10. 关键测试要点提示（容易漏的点）

以下是基于 PRD 容易被忽视、但出错代价大的检查项，请重点关注：

### 10.1 累计进位法（PRD 3.7）
- **必算样例**：200 股 / 24 期 / 6 月 cliff / 月度
- **必查**：第 1 期 cliff 日的归属应是 50 股（200×6/24，正好整除）
- **必查**：每期归属数 + 累计归属数 都要校验
- **必查**：最后一期归属数等于剩余余数，确保总和 = 200
- **变体**：100 股 / 3 期 必出现 33 + 33 + 34 模式（或类似）

### 10.2 FIFO 行权分配（PRD 3.8）
- **必查**：归属记录按归属日升序消耗
- **必查**：跨多条记录拆分时每条记录的状态推进正确（Vested → Partially Settled → Settled）
- **必查**：行权后剩余 operableOptions 等于授予总数 - 已 Settled 数

### 10.3 Grant 状态机（PRD 4.1）
- **必查**：Draft → Granted 不能跳过
- **必查**：完全归属 + 全部结算 → All Settled，不可逆
- **必查**：Option Closing 期满 → 未行权全部清零，状态变 All Settled
- **必查**：RSU Closing 立即清零（不等窗口期）

### 10.4 数据隔离（PRD 7.3）
- **重点**：员工 A 通过任何 URL 拼接 / API 调用 / Cookie 篡改尝试访问员工 B 的数据，全部应 403/404
- **重点**：管理员只能看到自己用工主体下的数据（如果有用工主体隔离）

### 10.5 税务事件三态流转（PRD 4.6）
- 待支付 → 已上传凭证 → 已确认（已确认不可逆）
- RSU 自动生成（归属当日）vs Option 行权后生成
- 缺估值时不生成（PRD 4.4 + 10）

### 10.6 Maker-Checker（PRD 1.6.5）
- 创建者 ≠ 审批者
- 申请提交后不可由提交人自审
- 审批日志保留两人记录

## 11. 测试结束的判定标准

### 11.1 测试可以提交报告的条件
- 414 条用例已全部走过一遍（PASS / FAIL / BLOCKED / NEEDS_CLARIFICATION 都算"走过"）
- 所有 BUG 已记录并赋予严重等级
- 所有 CLARIFICATION 已记录
- 三份交付物（TEST_REPORT / BUG_LIST / CLARIFICATION_LIST）已生成
- 附件目录已整理

### 11.2 不能结束的条件（必须继续）
- 还有用例未尝试执行（BLOCKED 不算未尝试，但要记录被谁阻塞）
- 已发现的 S0 阻断 BUG 还在调查复现条件
- 覆盖率自检表有空格未填

### 11.3 测试不通过、需返工开发的判定（建议）
- 任何 S0 阻断 BUG 存在
- S1 严重 BUG ≥ 3 个
- 核心算法（累计进位 / FIFO / 状态机）出错
- 权限矩阵或数据隔离失效

## 12. 测试执行的伦理与原则

1. **只报告事实**：观察到什么记什么，不夸大不缩小
2. **不护短，不挑刺**：失败要报，过度严格的 UI 鸡蛋里挑骨头要克制
3. **PRD 是天，但 PRD 不是一切**：PRD 没说的不要瞎判 bug，记 NEEDS_CLARIFICATION
4. **可复现优先**：偶发问题尽量找规律，找不到就老实标 FLAKY
5. **保护测试数据**：测试中产生的数据如包含拟真的个人信息，结束后清理
6. **不修代码**：再次强调，**不修代码**

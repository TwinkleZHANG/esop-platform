# 股权激励管理系统 测试分析与用例文档 v2

> **文档版本**: v2
> **依据**: PRD v4 (2026-04-20)
> **测试性质**: 黑盒测试，仅基于 PRD 验证系统是否符合需求，不参考任何代码实现
> **测试目标**: 全面覆盖功能点、权限矩阵、边界条件、端到端业务流程、状态机流转、数据隔离

---

## 0. 测试总览

### 0.1 测试范围

本文档覆盖以下测试维度：

1. **认证与权限**：登录、首次改密、密码重置、四角色权限矩阵、路由保护、API 鉴权
2. **基础数据 CRUD**：激励计划、员工档案、用工主体、持股主体、估值记录
3. **授予管理与状态机**：Grant 生命周期、归属记录生成（累计进位法）、状态聚合
4. **核心算法**：归属数量计算、FIFO 行权分配、已授予数量计算、剩余额度
5. **业务流程**：RSU 归属交割、Option 行权、Post-settlement、审批、税务确认
6. **关闭流程**：管理员关闭 / 离职关闭 / Closing 窗口期到期
7. **税务事件**：自动生成、状态流转、凭证上传与确认、Excel 导出
8. **员工端**：申请提交、校验规则、数据隔离、提醒机制
9. **管理端 ↔ 员工端联动**：状态同步、自动刷新
10. **边界条件**：数量、日期、文件、文本长度、并发
11. **UI/UX 规范**：响应式布局、搜索筛选、状态持久化、按钮样式
12. **定时任务**：归属检查、Closing 窗口期检查、税务事件补生成

### 0.4 测试数据约定

测试需要的初始账号（PRD 7.1 节）：

## 1. 认证与首次登录（TC-AUTH）

### TC-AUTH-001 邮箱密码登录成功
- **描述**：使用正确邮箱与密码登录系统
- **前置**：用户已在系统中存在，密码已设置
- **输入**：在 /login 页面输入正确邮箱 + 正确密码 → 提交
- **预期**：登录成功，跳转到对应角色的首页（管理员到管理端 Dashboard，员工到员工端总览）

### TC-AUTH-002 错误密码登录
- **描述**：密码错误时不应允许登录
- **前置**：用户已存在
- **输入**：正确邮箱 + 错误密码 → 提交
- **预期**：登录失败，显示错误提示；不应跳转任何受保护页面

### TC-AUTH-003 不存在的邮箱
- **描述**：邮箱不存在时不应允许登录
- **输入**：未注册邮箱 + 任意密码 → 提交
- **预期**：登录失败，错误提示；不应泄露"用户不存在"等过于精确的信息（建议返回通用"邮箱或密码错误"）

### TC-AUTH-004 首次登录强制改密码（种子超级管理员）
- **描述**：从环境变量种子生成的初始超级管理员首次登录必须强制改密码
- **前置**：种子脚本运行后，系统中存在初始超管账号，`mustChangePassword = true`
- **输入**：使用 ADMIN_EMAIL + ADMIN_INITIAL_PASSWORD 登录
- **预期**：登录成功后强制跳转到 /change-password 页面，无法访问其它任何页面

### TC-AUTH-005 首次登录强制改密码（管理员添加的员工）
- **描述**：管理员在员工档案添加员工后，员工首次登录强制改密
- **前置**：管理员已添加员工 A 并发送初始密码
- **输入**：员工 A 使用初始密码登录
- **预期**：登录后强制跳转到 /change-password；改密前无法访问员工端其它页面

### TC-AUTH-006 改密成功后正常使用
- **描述**：完成强制改密后 mustChangePassword 应被置为 false
- **前置**：用户首次登录已被引导到改密页面
- **输入**：在改密页面输入新密码（满足强度要求）→ 提交 → 重新登录
- **预期**：第二次登录直接跳转到对应首页，不再被强制改密

### TC-AUTH-007 密码重置后再次强制改密
- **描述**：超级管理员重置员工密码后，员工下次登录被强制改密
- **前置**：员工 A 已正常使用系统；超管在用户管理页面对其执行"重置密码"
- **输入**：员工 A 使用新的临时密码登录
- **预期**：登录后强制跳转到 /change-password 页面

### TC-AUTH-008 未登录访问受保护页面
- **描述**：未登录用户访问任何业务页面应被重定向
- **输入**：清空 Cookie，直接访问 /admin/dashboard、/admin/plans、/employee/overview 等
- **预期**：所有受保护页面均跳转到 /login

### TC-AUTH-009 注销后重新访问
- **描述**：注销后访问受保护页面应跳转到登录
- **前置**：用户已登录
- **输入**：执行注销 → 访问 /admin/grants 或 /employee/grants
- **预期**：跳转到 /login

### TC-AUTH-010 Session 过期处理
- **描述**：长时间未操作或 Session 失效后，访问受保护页面跳回登录
- **输入**：Session 失效后再访问任何页面
- **预期**：跳转到 /login，已填写但未提交的表单数据可能丢失（提示用户重新登录）

## 2. 权限矩阵与路由保护（TC-PERM）

> 依据 PRD 7.2 权限矩阵。每条用例分别用 4 个角色账号执行，验证有权限者通过、无权限者被拒。

### TC-PERM-001 创建激励计划权限
- **描述**：仅超管和授予管理员可创建激励计划
- **输入**：分别用 4 种角色尝试访问"创建计划"按钮，并直接 POST 到 /api/plans 创建接口
- **预期**：超管 ✅、授予管理员 ✅、审批管理员 ❌、普通员工 ❌；前端按钮对无权角色不显示，API 对无权调用返回 403/401

### TC-PERM-002 审批激励计划权限
- **描述**：仅超管和审批管理员可审批计划
- **输入**：4 种角色尝试将"审批中"计划变为"已通过"
- **预期**：超管 ✅、授予管理员 ❌、审批管理员 ✅、普通员工 ❌

### TC-PERM-003 添加员工 / 持股主体 / 估值记录权限
- **描述**：所有管理员均可添加员工、持股主体、估值记录（PRD 1.3 注 2）
- **输入**：4 种角色尝试 +新增 操作
- **预期**：超管 ✅、授予管理员 ✅、审批管理员 ✅、普通员工 ❌

### TC-PERM-004 创建授予权限
- **描述**：仅超管和授予管理员可创建授予
- **输入**：4 种角色尝试创建授予
- **预期**：超管 ✅、授予管理员 ✅、审批管理员 ❌、普通员工 ❌

### TC-PERM-005 推进 Grant 状态权限（Draft → Granted）
- **描述**：仅超管和审批管理员可推进 Draft → Granted
- **输入**：4 种角色尝试将 Draft 状态 Grant 推进
- **预期**：超管 ✅、授予管理员 ❌、审批管理员 ✅、普通员工 ❌

### TC-PERM-006 关闭 Grant 权限
- **描述**：仅超管和审批管理员可关闭 Grant
- **输入**：4 种角色尝试关闭某个非终态 Grant
- **预期**：超管 ✅、授予管理员 ❌、审批管理员 ✅、普通员工 ❌

### TC-PERM-007 审批员工申请权限
- **描述**：仅超管和审批管理员可审批员工申请
- **输入**：员工已提交申请，4 种角色查看是否能看到审批按钮并通过/驳回
- **预期**：超管 ✅、授予管理员 ❌、审批管理员 ✅、普通员工 ❌

### TC-PERM-008 设置员工离职权限
- **描述**：仅超管和审批管理员可将员工设为离职
- **输入**：4 种角色尝试修改员工状态
- **预期**：超管 ✅、授予管理员 ❌、审批管理员 ✅、普通员工 ❌

### TC-PERM-009 确认税务事件权限（Maker-Checker）
- **描述**：仅超管和审批管理员可将税务事件从"已上传凭证"确认为"已确定"。授予管理员明确不可（PRD 1.3 注 1）
- **输入**：在已上传凭证状态的税务事件上，4 种角色尝试点击"确认"
- **预期**：超管 ✅、授予管理员 ❌、审批管理员 ✅、普通员工 ❌

### TC-PERM-010 用户管理权限（仅超管）
- **描述**：仅超级管理员可访问用户管理页面、修改角色、重置密码
- **输入**：4 种角色访问 /admin/user-management 及其 API
- **预期**：超管 ✅；其它三种角色访问页面显示"无权限"或重定向，API 返回 403

### TC-PERM-011 员工端访问 /admin/* 路由
- **描述**：普通员工无法访问任何管理端路由
- **输入**：员工尝试访问 /admin/dashboard、/admin/plans、/admin/grants、/admin/user-management 等
- **预期**：所有管理端路由均拒绝访问（重定向到员工端或显示无权限）

### TC-PERM-012 管理员访问 /employee/* 路由
- **描述**：管理员可访问员工端路由（PRD 7.3）
- **输入**：超管、授予管理员、审批管理员分别访问 /employee/overview
- **预期**：均可访问，且看到的是该管理员账号自己（如该管理员同时是员工/有授予）的数据

### TC-PERM-013 用户管理路由仅超管可访问
- **描述**：/admin/user-management/* 路由对非超管不可达
- **输入**：授予管理员、审批管理员访问 /admin/user-management
- **预期**：返回 403 或重定向到 dashboard，不应显示页面内容

### TC-PERM-014 API 端点权限校验（绕过前端）
- **描述**：直接调用 API 应当受到与前端按钮一致的权限保护
- **输入**：用普通员工账号的 Session/Token 直接 POST /api/plans、PATCH /api/grants/:id（推进状态）等
- **预期**：所有写操作均返回 403/401，不应有"前端隐藏按钮但 API 不校验"的漏洞

### TC-PERM-015 员工端 API 数据隔离
- **描述**：员工端 API 只返回该员工自己的数据（PRD 7.3、12.3）
- **输入**：员工 A 登录，调用 /api/grants 等列表 API；尝试通过 ID 直接访问员工 B 的 Grant、TaxEvent、OperationRequest 详情
- **预期**：列表只包含 A 的数据；直接访问 B 的资源 ID 返回 403/404，绝不能返回 B 的数据

### TC-PERM-016 角色变更即时生效
- **描述**：超管修改某用户角色后，该用户刷新页面菜单和权限立即更新（PRD 4.8）
- **输入**：超管将用户 X 从员工改为审批管理员 → X 刷新页面
- **预期**：X 的侧边栏从员工端切换为管理端；X 立即拥有审批权限

## 4. 激励计划池（TC-PLAN）

### TC-PLAN-001 创建 RSU 计划 - 必填校验
- **描述**：必填字段为空时不可提交
- **输入**：将"计划标题"留空 → 提交
- **预期**：表单校验失败，提示该字段必填，不发送请求

### TC-PLAN-002 创建 RSU 计划 - 交割方式必选多选
- **描述**：RSU 类型下交割方式为多选（实股 / LP份额 / 境外SPV股份），至少选一个
- **输入**：选择 RSU 类型，不勾选任何交割方式 → 提交
- **预期**：校验失败，提示至少选择一项

### TC-PLAN-003 创建 RSU 计划 - 交割方式多选生效
- **描述**：可同时勾选多个交割方式
- **输入**：勾选实股 + LP份额 → 提交
- **预期**：创建成功，详情页显示两个交割方式

### TC-PLAN-004 创建 Option 计划 - 交割方式固定
- **描述**：Option 类型下交割方式固定为"购买实股的权利"，不可编辑
- **输入**：选择 Option 类型
- **预期**：交割方式字段显示固定文案，不可勾选/输入

### TC-PLAN-005 创建计划 - 适用法域必选
- **描述**：适用法域必选，三选一
- **输入**：依次创建三个计划，分别选内地 / 香港 / 海外
- **预期**：均可成功创建，详情页正确显示

### TC-PLAN-006 创建计划 - 激励池规模为 0
- **描述**：激励池规模必填，0 为非法值
- **输入**：激励池规模填 0 → 提交
- **预期**：校验失败或后续无法创建任何授予（剩余额度为 0）

### TC-PLAN-007 创建计划 - 激励池规模为负数
- **描述**：激励池规模不应允许负数
- **输入**：激励池规模填 -100
- **预期**：校验失败，提示需为正数

### TC-PLAN-008 创建计划 - 激励池规模超大数
- **描述**：超大数（如 9999999999）应可正常存储
- **输入**：激励池规模填 9999999999
- **预期**：创建成功，列表与详情页正确展示

### TC-PLAN-009 创建计划 - 生效日期默认值
- **描述**：生效日期默认为创建当日
- **输入**：打开创建表单
- **预期**：生效日期字段已预填为当天

### TC-PLAN-010 创建计划 - 生效日期可改
- **描述**：可手动修改生效日期为过去或未来
- **输入**：将生效日期改为 2020-01-01 或 2030-01-01
- **预期**：均可保存

### TC-PLAN-011 创建后默认状态
- **描述**：创建成功后状态默认为"审批中"
- **输入**：授予管理员创建计划
- **预期**：列表显示该计划状态为"审批中"

### TC-PLAN-012 审批管理员审批通过
- **描述**：审批管理员可将"审批中"变为"已通过"
- **输入**：审批管理员进入审批中计划详情 → 点击"审批通过"
- **预期**：状态变为"已通过"；列表对应行状态更新

### TC-PLAN-013 不设驳回操作
- **描述**：审批界面无"驳回"按钮（PRD 4.1 状态流转）
- **输入**：审批管理员查看审批中计划
- **预期**：仅有"审批通过"或"返回列表"按钮，无驳回选项

### TC-PLAN-014 已通过状态不可再变更
- **描述**：已通过状态无法回滚为审批中或其它状态
- **输入**：尝试通过任何方式（API 或 UI）将已通过计划改回审批中
- **预期**：操作被拒绝

### TC-PLAN-015 仅授予管理员可编辑审批中计划
- **描述**：审批中状态下只有授予管理员或超管可编辑

## 10. 归属计算（累计进位法）与 FIFO（TC-VEST）

### TC-VEST-001 累计进位法 - 200 股 24 期 6 月 cliff
- **描述**：基于 PRD 5.2 节示例验证累计进位法
- **输入**：totalQuantity 200，归属年限 2 年，cliff 6 月，按月归属（即 cliff 一次 + 18 期月度，但 PRD 示例为 24 期总数，理解为悬崖期合并到第 6 期）

  按 PRD 表述：第 6 期合并 cliff 部分，第 7-24 期月度（即从 vestingStartDate 起算共计 24 个月，第 6 个月是首次归属）
- **预期**：
  - 第 6 期：round(200 × 6/24) = 50
  - 第 7 期：round(200 × 7/24) - 50 = 8
  - 第 8 期：round(200 × 8/24) - 58 = 9
  - 第 9 期：round(200 × 9/24) - 67 = 8
  - ……
  - 最后一期补齐到 200

### TC-VEST-002 累计进位法 - 1200 股 1 年期 6 月 cliff 按月
- **描述**：PRD 4.5 注：第 6 月 600，第 7-11 月各 100，第 12 月补齐
- **输入**：totalQuantity 1200，年限 1 年，cliff 6 月，按月
- **预期**：共 7 条归属记录：[600, 100, 100, 100, 100, 100, 100]，总和 = 1200

### TC-VEST-003 累计进位法 - 总量恰好等于 totalQuantity
- **描述**：所有归属记录数量之和必须精确等于 totalQuantity（PRD 5.2 + 4.5）
- **输入**：极端值 totalQuantity = 7（不能整除任何归属期）
- **预期**：和恰好为 7，最后一期吸收余数

### TC-VEST-004 累计进位法 - 大数 totalQuantity
- **描述**：大数情况下精度不丢失
- **输入**：totalQuantity = 1234567，年限 4 年，cliff 12 月，按月（共 48 期，cliff 一次性归属前 12 期合并 → 第 12 期 + 13-48 期共 37 条）

- **输入**：totalQuantity 200，多次取不同期数关闭，对比已归属之和与 200 × n/24
- **预期**：差值绝对值 ≤ 1

### TC-VEST-006 累计进位法 - 0 cliff 按年归属

## 15. 端到端业务流程（TC-FLOW）

> 这一章是核心：完整跑通真实业务路径，验证多模块联动正确性。

### TC-FLOW-001 RSU 完整流程：从计划到 All Settled
- **描述**：完整跑 RSU 一个月度归属周期
- **前置**：超管账号、授予管理员、审批管理员、员工 A、估值（FMV=10）已存在
- **输入步骤**：
  1. 授予管理员创建 RSU 计划，池规模 10000，提交 → 审批中
  2. 审批管理员审批通过 → 已通过
  3. 授予管理员创建 Grant：员工 A，1200 股，1 年期，6 月 cliff，按月归属，2024-06-01 起算 → Draft
  4. 审批管理员补填协议ID，推进到 Granted → 自动生成 7 条 Pending 归属
  5. 时间推进至 2024-12-01：定时任务运行 → 第 1 条归属 Pending → Vested + 自动生成归属税务事件（待缴款）+ Grant → Vesting
  6. 员工 A 上传缴款凭证 → 税务事件 → 已上传凭证
  7. 审批管理员确认 → 税务事件 → 已确定 → 第 1 条归属 → Settled + Grant.operableShares = 600
  8. 后续 6 个月每月重复 5-7（直到所有 7 条归属都 Settled）
  9. 最后一笔确认后 Grant → All Settled
- **预期**：
  - 整个流程无错误
  - 每次确认后 operableShares 累计增加：600/700/800/900/1000/1100/1200
  - Grant 状态依次为 Vesting → Vesting → ... → Fully Vested →（在最后一笔归属确认时）All Settled
  - 状态变更日志完整记录每次变化（含操作者）
  - 计划池剩余额度始终为 10000 - 1200 = 8800（Grant 一直是非 Closed，按 totalQuantity 全额计入）
  - 员工端授予记录、归属详情、税务记录、总览资产汇总均与管理端一致

### TC-FLOW-002 Option 完整流程：行权后实股出售
- **描述**：完整跑 Option 行权 + 已行权实股的 post-settlement
- **前置**：Option 计划已通过、估值 FMV=10、员工 B
- **输入步骤**：
  1. 授予管理员创建 Option Grant：员工 B，1200 份，行权价 5 HKD，行权期 5 年，1 年期归属，6 月 cliff，按月，2024-06-01 起算 → Draft
  2. 审批管理员补填协议ID，推进 Granted → 7 条 Pending 归属，exerciseDeadline=2029-06-01
  3. 时间推进到 2024-12-01：定时任务 → 记录1 → Vested，exercisableOptions=600，Grant.operableOptions=600；Grant → Vesting
  4. 员工 B 提交行权 500 份的申请 → 待审批
  5. 审批管理员审批通过 → 自动生成行权税务事件（待缴款，FMV=10，strikePrice=5）
  6. 员工 B 上传凭证 → 已上传凭证
  7. 审批管理员确认 → 已确定 → FIFO 消耗记录1 500 份 → 记录1 → Partially Settled（exercisableOptions=100）；Grant.operableOptions=100，operableShares=500
  8. 时间推进到 2024-12-31：剩余 6 条归属陆续 Vested 后 Grant → Fully Vested → Still Exercisable
  9. 员工 B 对实股发起售出 200 → 审批通过 → 生成 post-settlement 税务（实股目标）→ 凭证 → 确认 → operableShares=300
  10. 员工 B 行权剩余 700 期权 → 审批 → 税务确认 → 全部记录 Settled → Grant → Still Exercisable → All Settled？等待
- **预期**：
  - 第 7 步后归属记录1 状态 Partially Settled，operableOptions=100，operableShares=500
  - 第 9 步税务事件 operationTarget=SHARES，operationType=售出
  - 直到所有归属 Settled 才进 All Settled

### TC-FLOW-003 RSU post-settlement 转让流程
- **描述**：RSU 已 Settled 实股转让
- **前置**：员工 C 名下 RSU Grant operableShares=500
- **输入**：
  1. 员工 C 提交转让 100 申请 → 待审批
  2. 审批管理员审批通过 → 生成 post-settlement 税务（operationTarget=实股或空，operationType=转让）
  3. 凭证 → 确认 → operableShares = 400
- **预期**：归属记录全程 Settled 不变；Grant operableShares 从 500 → 400

### TC-FLOW-004 Option 期权转让（FIFO）
- **描述**：Option 已归属未行权期权的转让，按 FIFO 消耗
- **前置**：Option Grant operableOptions=700（记录1 600，记录2 100）
- **输入**：
  1. 员工 D 提交转让期权 500 申请（目标：期权）
  2. 审批通过 → 生成 post-settlement 税务（operationTarget=OPTIONS，operationType=转让）
  3. 凭证 → 确认
- **预期**：FIFO 消耗：记录1 减 500 → exercisableOptions=100，状态 Partially Settled；记录2 不变；Grant.operableOptions = 200；operableShares 不变

### TC-FLOW-005 RSU Grant 关闭流程（Pending 归属仍存在）

  3. 员工窗口期内行权 500 → 审批通过 → 税务确认 → operableOptions=0，operableShares=700，所有相关归属记录 → Settled

## 23. PRD 模糊点 / 待澄清问题清单

> 测试执行中如发现以下模糊点，请在执行报告中标注，不直接判定为 bug：

1. **缺估值时归属记录是否变 Vested**：PRD 4.4 与 10 都说"不生成税务事件"，但未明确归属记录是先 Vested 后补税，还是连归属也延后。建议两种实现都可被接受，但需在执行报告中观察记录。

2. **Partially Settled 在 Closed 后的归属数量计入消耗**：PRD 4.1 表述 Option Closed 仅 Settled 计入消耗，那 Partially Settled 状态时的已行权部分（如归属记录 quantity=100，已行权 70，Closed 时状态仍 Partially Settled）这 70 是否计入"已消耗"？严格按 PRD 文字理解应不计入，因为只看 Settled 状态。

3. **Closing 窗口期内待审批申请的处理**：PRD 仅说窗口期到期时关闭待审批申请，未明确进入 Closing 时是否影响已有待审批申请。

4. **同一员工是否可有多角色**：PRD 表述每个用户一个角色（UserRole 枚举），但管理员是否可同时是被授予的员工（PRD 7.3"管理员可访问 /employee/* 路由"）需观察。

5. **角色降级唯一超管的处理**：PRD 未明确禁止将唯一超管降级，需观察实现。

6. **离职员工 operableShares 归零后是否仍可登录**：PRD 4.2"直至所有持有 operableShares 的 Grant 处理完毕"未明确处理完毕后账号去向。

7. **资产管理列表是否有"市值"列**：PRD 4.7 列表展示的表头不含市值，但数据规则中说"持股当前市值"，需观察实际列展示。

8. **同一估值日期多条记录的 FMV 选择**：PRD 4.4 说"日期最接近且不晚于触发日"，未规定同日多条估值时的处理。
